# GCデータ投入ガイド

## 2026-04-20 追加ルール（GC2014 / CBP）

- 対象は `machine_id=gc2014` / `column_id=cbp` を先行投入する。
- 同一機械の `CBP` と `CBP20` は同一カラム扱いとし、JSONでは `column_id=cbp` に統一する。
- 温度条件は以下を使用する。
  - `cbp_60c`
  - `cbp_80c`
  - `cbp_90c_2`
  - `cbp_120c`
  - `cbp_70c_isothermal`
- 90C(2) は現段階で1条件として `cbp_90c_2` に集約する。
- ソースは当面 `source=manual_scan_pdf` で管理する。

## 確度（certainty）

- `high`: 印字で明確。
- `medium`: 手書きだが比較的判読可能。
- `low`: 手書きで不確か。

## 名称正規化

- `analyte_original` には元データの表記を保持する。
- `analyte_normalized` には正規化名を入れる。
- `ブロワ` 系の名称は採用しない。該当レコードは `analyte_normalized="名称未確定"` にする。
- `名称未確定` はRT一覧には表示するが、候補提案ロジックの主対象から除外する。

## 候補提案ロジック（運用）

- coverage（分析対象のカバー件数）を最優先。
- 最小RT差が小さい条件は減点。
- `low` / `medium` を含む条件は減点（lowをより強く減点）。
- `名称未確定` を含む条件は減点し、理由欄で「未確定データを含む」を明示する。
- UI文言は「最適条件」ではなく「候補提案」を使う。

## 将来拡張

- 17A / 14B は未投入でも壊れないよう、`machine_id` と `column_id` を固定値にせず追加可能な配列構造を維持する。
- 将来的なCSV→JSON変換に備え、1行1レコードのフラット構造（records配列）を維持する。
